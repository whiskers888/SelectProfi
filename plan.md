## Актуализированный план реализации

### 1. Архитектурная цель

Целевая система: набор доменных сервисов (`Auth`, `Profile`, `Vacancy`, `Search`, `Application`, `Messaging`, `Payment`, `Notification`, `Analytics`, `Admin`) с выделением в микросервисы по мере роста нагрузки и команд.

Старт реализации: модульный монолит в текущем backend-репозитории с четкими границами доменов и контрактами интеграции. Это снижает стоимость старта и позволяет перейти к микросервисам без переписывания доменной логики.

Технологический baseline:

- `.NET 10` (синхронно текущему проекту)
- `PostgreSQL` как основная транзакционная БД
- `Redis` для кэша и распределенных блокировок
- `RabbitMQ` для доменных событий и асинхронных интеграций
- `Elasticsearch` для поиска (подключается на этапе Search)
- `MongoDB` только для чатов/сообщений (этап Messaging)
- `REST + OpenAPI`

### 2. Этапы backend (итеративно, от базового к расширению)

#### Этап B0. Платформенный фундамент

Цель: подготовить стабильный каркас проекта, на который дальше безопасно наращивается функциональность.

Объем:

- Базовая структура решения по модулям и слоям
- Конфигурация окружений (`Development`, `Staging`, `Production`)
- Подключение PostgreSQL + миграции
- Единый формат ошибок и валидации запросов
- OpenAPI для всех endpoint-ов
- Docker Compose для локального запуска backend + инфраструктуры

Критерии готовности:

- Приложение поднимается локально одной командой
- Есть `health` endpoint и проверка доступности зависимостей
- Есть первая миграция и успешный запуск БД

#### Этап B1. Auth + базовый Profile

Цель: получить рабочий контур пользователей и ролей.

Объем:

- Регистрация/логин, refresh-token, роли
- Базовый профиль пользователя (`customer`, `executor`, `candidate`, `admin`)
- Авторизация endpoint-ов по ролям
- Аудит ключевых действий (минимальный)

Критерии готовности:

- Пользователь проходит полный auth-flow
- Доступ к защищенным endpoint-ам корректно ограничен ролями
- OpenAPI покрывает auth/profile API

#### Этап B2. Vacancy + Application (MVP)

Цель: собрать основной бизнес-поток «публикация заказа -> отклик».

Объем:

- CRUD заказов и управление статусами
- Создание откликов и смена статусов отклика
- Базовые ограничения прав на действия

Критерии готовности:

- Заказчик создает и публикует заказ
- Исполнитель откликается на заказ
- История статусов сохраняется корректно

#### Этап B3. Search (индексация + фильтры)

Цель: быстрый и масштабируемый поиск по заказам и профилям.

Объем:

- Индексация данных в Elasticsearch через события RabbitMQ
- Фильтры и сортировки по ключевым полям
- Пагинация и релевантность

Критерии готовности:

- Данные попадают в индекс после изменений в доменных модулях
- Поиск возвращает стабильные и релевантные результаты

#### Этап B4. Messaging + Notification

Цель: коммуникации и уведомления для операционных сценариев.

Объем:

- Диалоги и сообщения (MongoDB)
- Доставка уведомлений по событиям
- Базовые шаблоны уведомлений

Критерии готовности:

- Сообщения сохраняются и читаются из диалогов
- Уведомления формируются по ключевым событиям (новый отклик, изменение статуса)

#### Этап B5. Payment + модерация + аналитика

Цель: завершить коммерческий контур и админ-функции.

Объем:

- Платежная интеграция и обработка webhook
- Инструменты модерации админа
- Базовые операционные отчеты

Критерии готовности:

- Платежный цикл проходит от инициализации до финального статуса
- Админ выполняет критические операции из API
- Доступны базовые метрики по заказам/платежам

### 3. Этапы frontend (после стабилизации backend-контрактов B1-B2)

Технологии:

- `React 18`, `TypeScript`, `Vite`
- `Redux Toolkit + RTK Query`
- Генерация клиента из backend OpenAPI

Последовательность:

- F0: инициализация приложения и базовый layout
- F1: auth + профиль + роль-ориентированные маршруты
- F2: заказы и отклики
- F3: поиск и фильтры
- F4: чаты, уведомления, платежные интерфейсы
- F5: административный интерфейс и отчеты

### 4. Сквозные практики качества (обязательные с B0)

- Unit + integration тесты для каждого нового модуля
- Контрактная проверка OpenAPI в CI
- Линтеры и форматирование в pipeline
- Наблюдаемость: структурные логи, метрики, трассировка
- Минимум один e2e сценарий на критический бизнес-путь к концу B2

### 5. Что делать прямо сейчас

Следующий шаг реализации: `Этап B0`.

Первый подшаг B0: привести текущий backend-каркас к production-ready baseline:

- убрать шаблонный `weatherforecast`
- добавить `health` endpoint
- подключить конфигурацию БД и базовый слой инфраструктуры
- подготовить основу для миграций
