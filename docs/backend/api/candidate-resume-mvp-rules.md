mc# Candidate Resume MVP Rules

Дата фиксации: 2026-04-03  
Статус: updated baseline требований (архитектурная фиксация перед серверной реализацией)

## 1. Целевой бизнес-поток

1. `Customer` создает `Order` с кратким описанием требуемого специалиста.
2. Несколько `Executor` откликаются на `Order`.
3. `Customer` выбирает одного `Executor`.
4. `Customer` и выбранный `Executor` согласуют требования вакансии через чат (видео-встречи вне MVP).
5. `Executor` создает `Vacancy` как черновик и отправляет на согласование.
6. После подтверждения от `Customer` вакансия публикуется и становится видимой кандидатам.
7. `Executor` проводит интервью, ведет кандидатов в `pool`, формирует `shortlist`.
8. `Customer` выбирает одного финального кандидата из `shortlist`.

## 2. Модель кандидатов в вакансии

- `pool` — рабочий список кандидатов по вакансии.
- `shortlist` — кандидаты из `pool`, отправленные заказчику.
- `selectedCandidateId` — один финальный выбор `Customer` из `shortlist`.

Инварианты:

- Кандидат может попасть в `shortlist` только из `pool` той же вакансии.
- `selectedCandidateId` обязан ссылаться на кандидата из `shortlist` той же вакансии.
- На одну вакансию допускается только один финально выбранный кандидат.

## 3. Источники кандидата и резюме

- `RegisteredUser`: кандидат уже зарегистрирован в системе.
- `AddedByExecutor`: кандидат и/или резюме впервые заведены рекрутером вручную.

Правила:

- Для `RegisteredUser` нельзя создавать дублирующий профиль кандидата.
- Для `AddedByExecutor` допускается создание кандидата и резюме для работы по вакансии.
- Рекрутер может добавлять кандидата в вакансию из общей базы или вручную с внешних площадок.

## 4. Контакты кандидата и доступ к ним

К контактам относятся: `email`, `phone`, ссылки на соцсети/мессенджеры.

Правила доступа:

- Рекрутер, который создал кандидата или первым привязал его к вакансии, получает доступ к контактам на 180 дней.
- Доступ рекрутера к контактам ограничен рабочим контекстом вакансии и TTL 180 дней.
- Вторые и последующие рекрутеры не получают доступ к контактам кандидата ни на каком этапе.
- Другие кандидаты и посторонние пользователи не получают прямой доступ к контактам.
- `Customer` получает доступ к полным контактам только после финального выбора кандидата по вакансии.

Правила отображения профиля в листингах и поиске:

- Для пользователей без права на контакты ФИО не показывается.
- В интерфейсах используется обезличенный алиас формата `Прилагательное + Специализация` (например, `Неопознанный аналитик`).
- Отображение только имени вместо алиаса в MVP не используется.

## 5. Уведомления кандидата

- При добавлении кандидата в процесс отправляется email-уведомление о приглашении.
- При назначении интервью отправляется email-уведомление.
- Если кандидат зарегистрирован, уведомления дублируются в системный inbox.
- Канал email обязателен для зарегистрированных и незарегистрированных кандидатов.

## 6. Комментарии рекрутеров и коммерциализация

- Каждый рекрутер может оставлять собственные комментарии по кандидату.
- Комментарии версионируются по автору и времени, без перезаписи истории.
- Непрошедшие кандидаты могут быть доступны другим рекрутерам через механизм выкупа данных.

Модель выкупа:

- Базовый актив: резюме кандидата (без контактов, если нет отдельного права).
- Дополнительный актив: комментарии конкретных рекрутеров.
- При выкупе у нового продавца покупатель докупает только недостающие комментарии/пакеты, без повторной покупки уже купленного базового резюме.
- Выкуп данных не открывает контакты кандидата второму и последующим рекрутерам.

## 7. Запрет повторной покупки

Инварианты покупки:

- Один и тот же покупатель не может повторно купить тот же базовый актив резюме кандидата.
- Если базовый актив уже куплен, покупка «расширенной версии» у другого рекрутера запрещена как дубликат базовой части.
- Разрешена только покупка новых комментариев, которых у покупателя еще нет.

Техническая фиксация (для реализации БД/API):

- Уникальность покупки по ключу `buyerId + candidateId + assetType(base_resume)`.
- Отдельная уникальность для комментариев по ключу `buyerId + candidateId + sellerId + assetType(comments)`.

## 8. Связывание профиля после регистрации кандидата

Когда кандидат, ранее добавленный рекрутером, регистрируется:

- Система создает `resume ownership proposal` кандидату.
- Если кандидат принимает предложение, управление резюме переходит кандидату.
- Если кандидат отклоняет предложение, резюме остается в рекрутерском контуре и может участвовать в коммерческой выдаче (по правилам маркетплейса).

Зафиксированное решение по неактивности:

- После принятия кандидатом ownership резюме не возвращается рекрутеру автоматически по таймауту неактивности.
- Причина: защита прав кандидата на персональные данные и предсказуемость владения.

## 9. Матчинг «резюме рекрутера» и «новый пользователь»

Базовый алгоритм матчинга:

- Primary key для поиска: нормализованные `FullName + BirthDate`.
- Secondary сигналы: `email`, `phone`.
- Автослияние без подтверждения кандидата запрещено; только предложение на подтверждение.

## 10. Материалы для заказчика

До финального выбора:

- Структурированное резюме.
- Комментарии и заметки интервью.
- Ответы на вопросы отбора.
- Контакты скрыты.

После финального выбора кандидата:

- Заказчику открываются полные контакты выбранного кандидата.

## 11. PDF и безопасность

- Экспорт в `PDF` разрешен только в обезличенном формате.
- В `PDF` исключаются персональные идентификаторы: ФИО, телефон, email, ссылки на соцсети/мессенджеры.
- Полная защита от скриншотов технически не гарантируется и фиксируется как ограничение платформы.

## 12. Архитектурные задачи для колонки `Architecture` в YouGile

- Ввести модель прав доступа к контактам кандидата с TTL 180 дней и аудитом выдачи.
- Разделить активы кандидата на `base resume` и `recruiter comments` с поддержкой частичного выкупа.
- Добавить реестр покупок с анти-дубли инвариантами.
- Добавить flow подтверждения ownership при регистрации кандидата.
- Добавить матчинг кандидатов по `FullName + BirthDate` с secondary-проверками `email/phone`.
- Зафиксировать событие открытия контактов заказчику только после выбора финального кандидата.

## 13. Open Questions (точки для отдельного решения до API)

- Финальная матрица монетизации: стоимость base-резюме, стоимость комментариев, правила выплат первичному автору.

## 14. Временная заглушка уведомлений (до выбора провайдера)

- Вводится интерфейс уведомлений с событиями `CandidateInvited` и `InterviewScheduled`.
- Для MVP используется `stub`-реализация: событие пишется в application log и технический outbox без внешней отправки.
- После выбора провайдера заглушка заменяется адаптером, контракт событий не меняется.

## 15. План подключения endpoint-ов к фронту (MVP Integration Plan)

Цель: подключать фронт по жизненному циклу `Order -> Vacancy(Status) -> Candidate Pipeline -> Contacts`, не дублируя бизнес-валидации backend.

### 15.1. Последовательность внедрения

1. Базовый API-клиент и единый parser ошибок `application/problem+json` (`status`, `code`, `detail`, `errors`).
2. Интеграция `Orders` для роли `Customer`:
   - `GET /api/orders`
   - `GET /api/orders/{orderId}`
   - `POST /api/orders`
   - `PATCH /api/orders/{orderId}`
   - `DELETE /api/orders/{orderId}`
   - Назначение рекрутера выполняется через `PATCH /api/orders/{orderId}` с полем `executorId`.
3. Интеграция `Vacancies`:
   - `GET /api/vacancies`
   - `GET /api/vacancies/{vacancyId}`
   - `POST /api/vacancies`
   - `PATCH /api/vacancies/{vacancyId}`
   - `PATCH /api/vacancies/{vacancyId}/status`
4. Интеграция candidate pipeline (основной контур):
   - `POST /api/vacancies/{vacancyId}/candidates/resumes`
   - `POST /api/vacancies/{vacancyId}/candidates/{candidateId}`
   - `PATCH /api/vacancies/{vacancyId}/candidates/{candidateId}/stage`
   - `PATCH /api/vacancies/{vacancyId}/selected-candidate`
5. Интеграция контактов:
   - `GET /api/vacancies/{vacancyId}/candidates/{candidateId}/contacts` (executor, owner+TTL)
   - `GET /api/vacancies/{vacancyId}/selected-candidate/contacts` (customer после финального выбора)

### 15.2. Обязательные UI-правила

- Для pipeline-операций UI должен учитывать `vacancy.status`:
  - операции изменения pipeline доступны только при `Published`;
  - при серверном `code = vacancy_not_published` показывать доменное сообщение и не выполнять ретрай.
- Ролевые действия скрывать/блокировать в UI, но не считать это заменой серверной авторизации.
- До финального выбора кандидата контакты заказчику не показываются.

### 15.3. Обработка ошибок (frontend contract)

- Использовать `code` из `ProblemDetails` как единственный источник для доменных алертов.
- Минимальный набор кодов для первого этапа:
  - `vacancy_not_published`
  - `candidate_not_in_shortlist`
  - `candidate_select_conflict`
  - `vacancy_candidate_contacts_access_denied`
  - `vacancy_access_forbidden`
  - `order_access_forbidden`

### 15.4. Ограничения текущего этапа

- Выбор рекрутера реализован в рамках существующего `PATCH /api/orders/{orderId}` (через `executorId`), отдельный endpoint для этого шага не требуется.
- Отдельный процесс «отклики нескольких рекрутеров» (лист откликов, статусы отклика, автотрекинг) пока не выделен в самостоятельный API-контур и остается следующим этапом.
