# Ручная проверка потока Order → Vacancy → Candidate

Дата: 2026-04-13

## Короткий ответ на вопрос про публикацию вакансии для соискателей

Да, на уровне backend вакансия публикуется для соискателей.

- Роль `Applicant` имеет доступ к чтению вакансий.
- Для `Applicant` доступны только вакансии со статусом `Published`.
- У роли `Applicant` есть backend endpoint для отклика: `POST /api/vacancies/{vacancyId}/respond`.

Но через текущий frontend соискатель не может полноценно откликнуться вручную на вакансию.

- В `Workspace` соискатель получает вакансии с сервера.
- При этом applicant-форма в UI пока не вызывает реальный backend endpoint отклика.
- То есть сценарий "соискатель увидел вакансию и нажал реальный отклик в интерфейсе" сейчас не доведён.
- Реальный отклик соискателя на текущий момент проверяется через API.

## Что проверено по окружению

- `docker compose up -d` в корне `Z:\SelectProfi` поднимает `app`, `postgres`, `redis`, `rabbitmq`.
- Backend доступен на `http://localhost:5268`.
- Автомиграции применяются на старте приложения.
- База в актуальном состоянии.
- `frontend` в docker-compose сейчас падает.
- Для ручной проверки frontend нужно запускать локально:

```powershell
cd Z:\SelectProfi\frontend
npm ci
npm run dev:server
```

## Рабочие учётки

Seed-аккаунты backend:

- `customer@selectprofi.local` / `1`
- `executor@selectprofi.local` / `1`

Для applicant backend endpoint есть, но повторный логин у новых зарегистрированных applicant-аккаунтов сейчас выглядит нестабильным, поэтому для ручной проверки applicant-flow лучше использовать текущую сессию сразу после `register` или проверять endpoint через API-клиент.

## Ручной сценарий через UI

Этот сценарий покрывает основной поток, кроме реального UI-отклика соискателя.

### 1. Заказчик создаёт заказ

1. Откройте frontend локально.
2. Войдите как `customer@selectprofi.local`.
3. Создайте заказ.
4. В поле "Требуемое количество кандидатов" укажите `3`.
5. Убедитесь, что заказ появился в списке.

Ожидаемый результат:

- заказ создаётся успешно;
- поле количества кандидатов сохраняется;
- заказ доступен исполнителям для отклика.

### 2. Исполнитель откликается на заказ

1. Выйдите из customer.
2. Войдите как `executor@selectprofi.local`.
3. Откройте заказ.
4. Выполните отклик на заказ.

Ожидаемый результат:

- отклик исполнителя сохраняется;
- заказчик видит отклик в деталях заказа.

### 3. Заказчик выбирает исполнителя

1. Выйдите из executor.
2. Войдите как `customer@selectprofi.local`.
3. Откройте тот же заказ.
4. Выберите исполнителя из откликов.

Ожидаемый результат:

- у заказа появляется назначенный исполнитель;
- повторный произвольный выбор другого исполнителя недоступен.

### 4. Исполнитель создаёт вакансию

1. Выйдите из customer.
2. Войдите как `executor@selectprofi.local`.
3. Создайте вакансию по выбранному заказу.

Ожидаемый результат:

- вакансия создаётся в статусе `Draft`;
- вакансия связана с нужным заказом.

### 5. Публикация вакансии

1. Под executor переведите вакансию в `OnApproval`.
2. Выйдите из executor.
3. Под `customer@selectprofi.local` переведите вакансию в `Published`.

Ожидаемый результат:

- после публикации вакансия становится доступной для applicant на уровне API;
- только `Published`-вакансии видны соискателю.

### 6. Добавление кандидатов исполнителем

Так как applicant UI-отклик ещё не подключён, текущий ручной сценарий закрывается через исполнителя.

1. Войдите как `executor@selectprofi.local`.
2. Откройте опубликованную вакансию.
3. Добавьте 3 кандидатов:
   - либо из системной базы;
   - либо вручную с резюме.
4. Для каждого кандидата переведите стадию в `Shortlist`.

Ожидаемый результат:

- первые 1–2 кандидата в `Shortlist` не должны выставлять отметку авто-отправки;
- после достижения порога `3` shortlist должен считаться автоматически отправленным заказчику.

### 7. Проверка авто-отправки shortlist

Проверьте:

- список вакансий;
- карточку вакансии.

Ожидаемый результат:

- до достижения порога отображается `Не отправлен` / `Не отправлено`;
- после достижения порога отображается `Отправлен (...)`.

### 8. Заказчик выбирает финального кандидата

1. Войдите как `customer@selectprofi.local`.
2. Откройте опубликованную вакансию.
3. Выберите одного кандидата из shortlist как финального.
4. Получите контакты выбранного кандидата.

Ожидаемый результат:

- заказчик может выбрать кандидата;
- контакты выдаются только для выбранного кандидата.

## Как проверить именно self-service отклик соискателя

Через UI этот сценарий сейчас не завершён.

Рабочий путь на текущий момент:

1. Зарегистрировать applicant через `POST /api/auth/register`.
2. Использовать полученный `accessToken`.
3. Вызвать `POST /api/vacancies/{vacancyId}/respond` с этим токеном.

Ожидаемый результат:

- backend создаёт или находит registered candidate;
- кандидат добавляется в pipeline вакансии на стадию `Pool`;
- повторный отклик в ту же вакансию блокируется.

## Фактический вывод

- Соискатели могут увидеть опубликованные вакансии.
- Соискатели могут реально откликнуться на опубликованную вакансию через backend API.
- Соискатели пока не могут выполнить этот отклик штатно через текущий frontend-интерфейс.
- Для полного UI-only сценария applicant-flow ещё нужен отдельный frontend-шаг доработки.
