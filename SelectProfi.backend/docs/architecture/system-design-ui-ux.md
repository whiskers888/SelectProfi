# System Design UI/UX Baseline (PRI-117)

Дата фиксации: 2026-03-28
Статус: Draft (рабочий стандарт до следующего ADR)

## 1. Область действия

Документ задает единые правила UI/UX для всех новых интерфейсных задач проекта.

- Платформы: `Web + Mobile`.
- Темы: `Light + Dark` обязательны для всех новых компонентов.
- Источник правды для токенов: `JSON Design Tokens`.
- Web-слой использует только сгенерированные `CSS Variables`.

## 2. Инженерные принципы

1. Консистентность: одинаковые задачи решаются одинаковыми паттернами.
2. Доступность: интерфейс обязан соответствовать минимуму `WCAG 2.2 AA`.
3. Масштабируемость: изменения в теме или бренде выполняются через токены, без точечных правок компонентов.
4. Платформенная совместимость: смысл и иерархия UI одинаковы на web и mobile.
5. Предсказуемость: у каждого интерактивного элемента есть видимая обратная связь по состояниям.

### 3.4 Интерактивные состояния

Для всех интерактивных компонентов:
- `default`
- `hover`
- `active`
- `focus-visible`
- `disabled`
- `selected` (где применимо)

Правила:
- `disabled` запрещает только действие, но сохраняет читаемость.
- `focus-visible` обязателен и не заменяется только цветовой сменой фона.
- Изменение состояния должно быть видно не только по цвету (иконка, бордер, подчеркивание или текст).

## 4. Design Tokens

### 4.1 Формат и source of truth

Единый источник: `JSON` (платформенно-нейтральный набор токенов).

Web:
- токены потребляются только как `CSS Variables`, сгенерированные из JSON.
- прямое использование hex в компонентах запрещено.

Mobile:
- токены импортируются из того же JSON через платформенный слой адаптации.

### 4.2 Именование токенов

Формат имени: `category.role.state.scale`.

Примеры:
- `color.bg.surface.default`
- `color.text.primary.default`
- `color.action.primary.hover`
- `space.4`
- `radius.md`
- `font.size.body.md`

Требования:
- имя токена описывает назначение, а не конкретный цвет.
- брендовые и семантические токены отделены от компонентных алиасов.

### 4.3 Минимальный состав токенов

- Color: brand, neutral, semantic, action, border, text, background, overlay.
- Typography: family, size, weight, lineHeight, letterSpacing.
- Spacing: базовая шкала.
- Radius: `sm/md/lg/xl/full`.
- Border: width/style.
- Shadow: `sm/md/lg`.
- Z-index: слой контента, overlay, modal, toast.
- Motion: duration/easing.

## 5. Типографика и layout

### 5.1 Типографическая шкала

- `display`: `48/56`, weight `700`
- `h1`: `36/44`, weight `700`
- `h2`: `30/38`, weight `700`
- `h3`: `24/32`, weight `600`
- `h4`: `20/28`, weight `600`
- `body-lg`: `18/28`, weight `400`
- `body-md`: `16/24`, weight `400`
- `body-sm`: `14/20`, weight `400`
- `caption`: `12/16`, weight `500`

На mobile допускается снижение display/h1 на один шаг, body-уровни сохраняются.

### 5.2 Сетка, отступы, брейкпоинты

- Базовый шаг отступа: `4px`.
- Стандартная шкала: `4, 8, 12, 16, 20, 24, 32, 40, 48, 64`.
- Radius по умолчанию: `12px` для surface-контейнеров, `10px` для controls.

Breakpoints:
- `mobile`: `0-767`
- `tablet`: `768-1199`
- `desktop`: `1200+`

## 6. Компонентные правила (базовый набор)

### 6.1 Button

- Варианты: `primary`, `secondary`, `ghost`, `danger`.
- Высота: `40px` desktop, `44px` mobile.
- У каждого варианта обязательны состояния из раздела 3.4.

### 6.2 Input / Select / Checkbox

- Видимый label обязателен (placeholder не заменяет label).
- Ошибка валидации: текст + иконка + цветовой акцент.
- Текст ошибки связан с полем через доступные атрибуты (aria-паттерн).

### 6.3 Modal

- Не более одного модального слоя поверх базового контента.
- Кнопка закрытия + закрытие по `Esc` (кроме критических подтверждений).
- Фокус-трап обязателен.

### 6.4 Table

- Горизонтальный скролл на narrow-screen обязателен.
- Состояния: loading, empty, error задаются отдельными шаблонами.

### 6.5 Toast

- Позиция web: правый верх/низ, не перекрывая critical-controls.
- Автозакрытие для info/success: `3-5` секунд.
- Error toast не исчезает мгновенно и должен иметь явное действие.

## 7. UX-поведение

1. Loading:
- Для операций до `400ms` допускается без спиннера.
- Для дольше `400ms` использовать skeleton или индикатор прогресса.

2. Empty state:
- Обязательны: причина пустоты + действие следующего шага.

3. Error state:
- Обязательны: что случилось, что делать дальше, кнопка retry при возможности.

4. Confirm patterns:
- Деструктивные действия подтверждаются отдельным диалогом.
- Текст подтверждения конкретный, без общих фраз.

5. Motion:
- Duration: `120ms`, `180ms`, `240ms`.
- Easing: `ease-out` для появления, `ease-in` для скрытия.
- Анимация не должна блокировать ввод и чтение.

## 8. Доступность (обязательные критерии)

1. Контраст текста к фону:
- обычный текст `>= 4.5:1`
- крупный текст `>= 3:1`

2. Контраст UI-элементов (бордеры/иконки/контролы) `>= 3:1`.
3. Все интерактивные элементы доступны с клавиатуры.
4. Видимый `focus-visible` обязателен.
5. Для медиа/анимаций учитывается `prefers-reduced-motion`.

## 9. Governance и изменения стандарта

1. Любое изменение токенов или UI-правил проходит через ADR-задачу.
2. Изменение палитры/семантики фиксируется версией дизайн-токенов.
3. UI-компоненты обновляются только после актуализации source JSON и генерации платформенных артефактов.
4. Нельзя держать локальные override-токены внутри feature-компонентов.

## 10. PR-чеклист соответствия UI/UX стандарту

1. Использованы токены, а не прямые hex/px-магические значения.
2. Реализованы состояния `hover/active/focus-visible/disabled`.
3. Проверены `light` и `dark` темы.
4. Проверены `mobile/tablet/desktop` брейкпоинты.
5. Контраст и keyboard-navigation не нарушены.
6. Empty/loading/error состояния реализованы и читаемы.
7. Для деструктивных действий есть confirm-паттерн.
8. Нет расхождения между JSON-токенами и платформенной реализацией.
